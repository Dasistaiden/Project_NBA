# Yahoo NBA Fantasy 選秀模擬器 — 專案規劃

> 承接 [01_value_proposition.md](01_value_proposition.md) 的範圍界定，本文件制定實作階段、MVP 定義與里程碑。文件結構比照 Project_finance 慣例。

## 1. 階段劃分 (Phases)

| 階段 | 名稱 | 目標 |
|---|---|---|
| Phase 1 | 資料 + 排名 MVP | 抓取 2025-26 球員數據入庫，計算可調權重的 Fantasy Point，Streamlit 依五大位置呈現排名 |
| Phase 2 | 健康度與角色定位 | 以過去 n 季出賽紀錄推導健康風險指數；以先發場次、上場時間、使用率推導角色定位 |
| Phase 3 | 新秀分析與心態 | 引入 NCAA/其他聯賽資料評估新秀；建立心態/質性評分欄位（人工維護） |
| Phase 4 | 選秀日輔助 | 選秀當下標記已被選走的球員、剩餘最佳人選即時推薦 |

Phase 1 就包含 Streamlit 介面 — 與 Project_finance 不同，本專案的 UI 是核心交付物而非加值功能，沒有介面就無法在選秀時使用。

## 2. Phase 1 MVP 範圍定義

**MVP 只做「數據進得來、分數算得出、位置看得到」。**

### 包含
- 資料抓取：以 `nba_api` 取得 2025-26 賽季已登錄球員名單與場均數據（一次性快照，手動更新即可 — 賽季已結束，資料不再變動）
- 資料儲存：SQLite 單檔資料庫
- Fantasy Point 計算：各項數據 × 權重加總；預設採 Yahoo Points League 標準權重（PTS +1, REB +1.2, AST +1.5, STL +3, BLK +3, TOV -1），權重可在介面上即時調整
- Streamlit 介面：五大位置（PG/SG/SF/PF/C）分頁，各位置依 Fantasy Point 排序，附上 2025-26 場均數據佐證

### 明確排除（留給後續 Phase）
- 健康度、角色定位、新秀、心態四項質性因子（Phase 2/3）
- Yahoo API 串接（官方位置資格認定需 OAuth，MVP 以 NBA 官方登錄位置對映五大位置替代）
- 每日排程更新（賽季已結束，無此需求；下賽季開打後若要追蹤再議）
- 選秀日互動功能（Phase 4）

### 已知限制（MVP 接受）
- NBA 官方位置只有 G / F / C / G-F / F-C 等粗分類，對映到 PG/SG/SF/PF/C 時一名球員可能同時出現在多個位置分頁 — 這與 Yahoo 的多位置資格（multi-position eligibility）行為類似，屬可接受近似

### MVP 完成定義（Definition of Done）
- 資料庫內有 2025-26 全部已登錄球員（約 450-550 名）的場均數據
- 抽查 3 名球星（不同位置）數據與 NBA 官網一致
- Streamlit 可切換五大位置、調整權重後排名即時變化

## 3. 里程碑清單 (Milestones)

### Phase 1：資料 + 排名 MVP
- [ ] M1.1 確認 `nba_api` 可取得球員名單、位置、場均數據（endpoint 與欄位驗證）
- [ ] M1.2 資料庫 schema 設計（players、player_stats）
- [ ] M1.3 抓取腳本完成，一次執行即完成全量入庫
- [ ] M1.4 Fantasy Point 計算模組（權重設定檔 + 計算函式）
- [ ] M1.5 Streamlit 介面：位置分頁 + 排名表 + 權重調整
- [ ] M1.6 對照 NBA 官網抽查驗證，達成 DoD

### Phase 2：健康度與角色定位
- [ ] M2.1 回補過去 3 季出賽紀錄（GP/GS/MIN）
- [ ] M2.2 健康風險指數計算邏輯（出勤率、傷病缺席趨勢）
- [ ] M2.3 角色定位標籤（先發/替補、球權使用率、進攻發起 vs 藍領）
- [ ] M2.4 介面整合：排名表加上健康/角色欄位

### Phase 3：新秀分析與心態
- [ ] M3.1 確認新秀 NCAA 資料來源（可行性研究）
- [ ] M3.2 新秀換算模型（NCAA 數據 → NBA 預期產出）
- [ ] M3.3 心態/質性評分欄位（人工維護的補充資料檔）

### Phase 4：選秀日輔助
- [ ] M4.1 已選走球員標記功能
- [ ] M4.2 剩餘最佳人選推薦

## 4. 資源預估

| 資源類型 | 內容 | 預估 |
|---|---|---|
| 人力 | 個人開發 | Phase 1 約 10-15 小時 |
| 資料源成本 | `nba_api`（免費，非官方封裝 NBA stats API） | 零成本；注意 stats.nba.com 有頻率限制，抓取需節流 |
| 運算/儲存 | 本機 SQLite | 一季 500 名球員數據 < 1MB，可忽略 |
| 依賴套件 | `nba_api`, `pandas`, `streamlit` | 皆免費開源 |

## 5. 風險與待確認事項 (Open Questions)

- `nba_api` 為非官方套件，stats.nba.com 偶爾封鎖雲端 IP 或改版；本機執行通常沒問題，M1.1 先驗證
- 球員位置對映的粗糙度是否影響實用性 → MVP 用過再決定是否需要外部位置資格來源
- Phase 3 新秀資料來源（NCAA API 或手動整理）可行性最不確定，屆時再細化
- 心態評分無客觀資料源，定位為「人工維護的備註欄位」而非自動計算，避免偽精確
